# Charter Purpose and Boundaries

A project charter is not the business case, project plan, backlog, architecture design, requirements document, or benefits realization plan.

## The charter does

- Authorize initiation
- Clarify sponsor and ownership
- State the business outcome being supported
- Define high-level scope boundaries
- Name authorized deliverables
- Establish constraints and assumptions
- Define success measures
- Identify risks, dependencies, and open decisions
- Establish governance, escalation, and change-control discipline
- Move the project into detailed planning and execution

## The charter does not

- Re-sell the project after approval
- Contain a full schedule baseline
- Replace detailed requirements
- Replace technical design
- Hide unresolved decisions behind optimistic prose
- Treat a broad transformation theme as a single governable project

## Boundary test

A charter is too broad when the sponsor cannot approve tradeoffs, the team cannot name deliverables, success measures are generic, or the project actually contains several independent initiatives.
