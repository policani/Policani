# Resume Evidence Engine

Resume Evidence Engine is a portable, evidence-bound job-search starter kit for Codex-style agents. It helps a user set up a resume evidence system, score job descriptions, create targeted resumes and cover letters, and hand final Markdown to an agent-managed DOCX renderer when the environment supports it.

The project is intentionally blank except for templates and fictional samples. It does not contain a user's private job history, credentials, preferences, references, or proof narratives.

## First-time user promise

Users should not need to install Git, Python, LibreOffice, PDF tools, or any local developer stack to get value from this project.

The core workflow is agent-readable Markdown:

1. Configure the project.
2. Build the user's factual evidence base.
3. Score a job description.
4. Create clean Markdown resume and cover-letter drafts.
5. Let Codex or another capable agent render DOCX only when its managed environment supports that.

If DOCX rendering is unavailable, the correct fallback is to provide the final Markdown and say that DOCX rendering could not be completed in the current environment. Do not ask first-time users to fix local Python packages.

## File strategy

Keep the working system flat and upload-friendly. Supporting documentation lives in `docs/`. Generated outputs and runtime caches should stay out of the repository.

## Complete file map

This section lists every source file that belongs in the beta package. Generated outputs and runtime caches are intentionally excluded.

### Project orientation and guardrails

- `.gitignore` - GitHub hygiene rules. Keeps generated outputs, Python bytecode, QA reports, renderer caches, and temporary files out of the public beta repository. Edit only when a new generated artifact type appears.
- `LICENSE` - PolyForm Noncommercial license for free noncommercial use. Keep this file in the public repository so users understand what they are allowed to do with the project and that commercial use requires separate permission.
- `README.md` - human-facing orientation for beta users and reviewers. Explains the project purpose, first-time setup, file map, DOCX handoff, maintenance model, and GitHub sharing rules. Keep this current whenever files, setup steps, or beta expectations change.
- `AGENTS.md` - compact operating instructions for Codex-style agents and the single source of truth for agent behavior. It tells the agent to load config first, stop on placeholders, require setup completion before normal use, score roles, customize evidence-bound materials, and render DOCX only when triggered. If another platform specifically expects `CLAUDE.md`, copy or rename this file for that environment instead of maintaining two instruction files in this repository.

### Public documentation

- `docs/workflow-flowchart.pdf` - lightweight one-page workflow diagram. It gives beta users a quick visual map of setup, scoring, evidence-bound writing, DOCX handoff, validation, and maintenance.
- `docs/readme.html` - browser-friendly version of this README with a more readable layout for beta users.
- `docs/github-upload-checklist.md` - beginner-friendly checklist for publishing this project to GitHub. It covers what to include, what not to include, suggested repository settings, and pre-release checks.
- `docs/sample-builder-outputs/` - fictional sample files generated in the same style as the browser builders. Use these as models for a completed core resume, resume variant, and case studies before replacing the sample details with real user facts.

### Browser builders

- `builders/config-builder.html` - local, no-server configuration form. A user opens it in a browser, answers controlled setup questions, previews the generated Markdown, and downloads `project-config.md`. It deliberately generates `Setup status: In Progress` because configuration is only the first part of setup. Edit when adding or changing project settings, scoring priorities, thresholds, hard stops, output defaults, or setup-plan fields.
- `builders/resume-builder.html` - local, no-server resume Markdown form. It can produce `core-canonical-resume.md`, `variant-resume-[role-family].md`, or `final-resume-[candidate-target].md` using project naming conventions. It is useful for users who prefer forms over hand-editing Markdown. It does not score jobs, tailor claims, or render DOCX.
- `builders/case-study-builder.html` - local, no-server proof narrative form. It produces Markdown case studies that follow `case-study-template-1.md` or `case-study-template-2.md`, with sample prompts that teach the user how to break a work story into context, stakes, choices, actions, outcomes, evidence quality, allowed claims, and guardrails.

### User configuration and setup

- `project-config.md` - compact value file generated by the config builder or edited during setup. Agents scan `Value:` and `Priority [number] value:` lines here before doing normal work. Any answer containing `Insert` blocks normal use. `Setup status` must be changed to `Complete` only after the final setup checklist is true. This file stores preferences and constraints, not work-history evidence.
- `setup-wizard.md` - guided first-time setup interview for an agent. It tells the agent what to ask, what files to update, and how to verify setup before regular scoring begins. Use it when the user asks for `/setup` or needs help completing the system. It is a process file, not a private evidence store.

### Evidence base and resume templates

- `core-canonical-resume.md` - factual master resume template. This should become the user's truth source for work history, dates, titles, scope, tools, outcomes, education, credentials, and claim guardrails. It should contain more evidence than any one final resume needs, but it must not include unsupported claims or private reference contact details.
- `variant-resume-template-1.md` - starter template for the user's first reusable role-family resume. Rename or replace it with a role-family-specific file such as `variant-resume-facilities-maintenance.md` once setup is underway. It should package the same truthful facts for one hiring lane.
- `variant-resume-template-2.md` - starter template for a second reusable role-family resume. Use it when the candidate has another credible lane that needs different positioning. Like the first variant, it should be renamed or replaced with a clear role-family filename.
- `evidence-map.md` - routing layer between job-description signals and proof. It should say which capabilities are supported, which source files prove them, which variant resume to use, which proof narrative is strongest, and what guardrails prevent exaggeration. This is one of the most important files for reliable scoring and customization.
- `case-study-template-1.md` - proof narrative template for a structured decision/action/outcome story. Use it to build a detailed case that supports resume bullets, cover letters, interview answers, and gap mitigation. Replace placeholders with real evidence, or duplicate the pattern into a new case file.
- `case-study-template-2.md` - proof narrative template with more emphasis on scale, operating context, measured/observed/inferred outcomes, and claim limits. Use it for complex business, risk, financial, operational, or transformation examples. Do not publish filled-in private case studies to a public beta repository.

### Writing and decision references

- `ref-scoring.md` - scoring rubric and output format. It defines pre-screening, fit dimensions, score bands, the proceed threshold, single-role output, batch output, and customization lift. Agents use it with `project-config.md` and `evidence-map.md` when evaluating job descriptions.
- `ref-writing.md` - resume-writing rules. It keeps materials ATS-friendly, evidence-bound, readable, and free of decorative layout, unsupported title inflation, and internal notes. Use it when creating or reviewing resumes and final Markdown.
- `ref-cover-letter-style-guide.md` - cover-letter rules. It defines structure, voice, length defaults, evidence expectations, openings, and gap handling. It keeps letters short, specific, and proof-based instead of generic or overly enthusiastic.
- `ref-credentials.md` - credential, education, license, certification, coursework, and training inventory template. It should eventually clarify what credentials exist, what they support, what they do not support, and where proof lives. Use it to avoid overstating qualifications.
- `ref-positioning.md` - gap and positioning reference. It records known concerns such as industry change, no degree, title mismatch, contractor history, compensation mismatch, or work-model constraints. It should define truthful framing, strongest mitigating evidence, and forbidden framing.

### DOCX renderer and handoff

- `md2docx-workflow.md` - agent-facing DOCX conversion workflow. It explains when rendering is triggered, what source Markdown must look like, what validation gates matter, and what to do when rendering is unavailable. It reinforces that conversion is a rendering step, not a rewriting step.
- `md2docx_resume_flat.py` - bundled Markdown-to-DOCX renderer for capable agent environments. It parses clean resume or cover-letter Markdown, applies Word styles, creates DOCX files, validates structure, and can package multiple outputs. It is included for agents to run when their managed runtime supports it; first-time users should not be asked to install Python packages.
- `renderer-config.json` - renderer defaults and validation settings. It points to `reference.docx`, sets the default output folder, controls fast versus QA behavior, defines style profiles, and stores validation rules. Edit only when document formatting, validation, or renderer defaults should change.
- `reference.docx` - Word style template used by the renderer. Keep it in the beta package. It is not a sample resume and not a generated output; it provides the base Word styles the renderer needs.
- `sample_resume.md` - fictional resume fixture for smoke testing the renderer and demonstrating the expected Markdown shape. It should remain fictional and safe to publish. Do not replace it with a real user's resume.
- `sample_cover_letter.md` - fictional cover-letter fixture for smoke testing the renderer and demonstrating the expected cover-letter Markdown shape. It should remain fictional and safe to publish.

The renderer files are included so Codex or a hosted agent can inspect and use them when execution is available. They are not intended to make first-time users install Python or debug a local runtime.

## Recommended first-time setup

1. Open `builders/config-builder.html`.
2. Complete the form and export `project-config.md`.
3. Replace the starter `project-config.md`.
4. Open `builders/resume-builder.html` when you want a form-based way to create `core-canonical-resume.md`, a variant resume file, or a final targeted resume `.md`.
5. Open `builders/case-study-builder.html` when you want a form-based way to create a proof narrative from a real work story.
6. Open `docs/sample-builder-outputs/` if you want to see plain completed examples before filling out your own files.
7. Copy `AGENTS.md` into the project instruction field if the platform requires pasted instructions. For a platform that specifically expects `CLAUDE.md`, use the contents of `AGENTS.md` there.
8. Start a new Codex chat in the project and ask for `/setup`.
9. Work through the setup interview to complete factual work history, evidence map, variants, credentials, positioning notes, and proof narratives.
10. Mark `Setup status` as `Complete` only after the final setup checklist is true.
11. Test the system with one job description using `/score`.

Do not run local install steps as part of first-time setup.

For a visual overview of the workflow, open `docs/workflow-flowchart.pdf`.

For a more readable browser version of this guide, open `docs/readme.html`.

For first-time GitHub publishing, use `docs/github-upload-checklist.md`.

## How DOCX handoff works

The engine writes final resumes and cover letters as Markdown first. DOCX conversion is a separate handoff step. When the agent has a supported execution environment, it may use the included renderer, `reference.docx`, and `renderer-config.json` to create a `.docx`.

Keep `reference.docx`. It is not a draft resume; it is the Word style template used by the renderer.

Rendering should happen when:

- the user asks for DOCX, Word, render, convert, final document, final resume, final cover letter, or download;
- `Default output workflow` is `Markdown first, then DOCX` and final resume or cover-letter Markdown exists;
- `Final resume DOCX handoff` is `Yes` and a final resume Markdown file has been generated;
- `Final cover letter DOCX handoff` is `Yes` and a final cover-letter Markdown file has been generated.

Rendering should not happen during scoring, brainstorming, gap analysis, or rough draft review unless the user explicitly asks.

PDF export is out of scope. Do not add a PDF step to the default workflow.

## Why the config builder exists

The configuration experience should be easy for a non-technical user. The HTML builder carries explanations and form controls. The exported `project-config.md` carries only the values the agent needs.

The engine scans `Value:` and `Priority [number] value:` lines. Any value containing `Insert` stops normal work and prompts the user to finish the missing setting.

The config builder sets `Setup status` to `In Progress`. This is intentional: configuration alone is not enough for scoring. Regular scoring and employer-facing drafting should begin only after the evidence system is complete and `Setup status` is changed to `Complete`.

The generated config also includes default scoring bands:

- Strong: 85-100
- Good fit: 75-84
- Proceed with caution: 65-74
- Stop: 0-64
- Proceed threshold: 65

## Resume file discovery

The system does not watch the folder in the background. Each time an agent works in the project, it reads the current files. New resumes and variants will be seen when they are saved in the project folder and follow the naming convention:

- `core-canonical-resume.md` for the factual master resume.
- `variant-resume-[role-family].md` for reusable role-family variants.
- `final-resume-[company-role].md` for one-job targeted resumes.

The resume builder follows these conventions when the filename is left blank. If a user adds many variants, keep role-family names clear and distinct so the agent can choose the right starting point.

## Evidence map and proof narratives

The evidence map is the routing layer. It should answer:

- What capability does a job description signal?
- Which resume variant should be used?
- Which proof narrative proves the claim?
- What claims are allowed?
- What guardrails prevent exaggeration?

Use the proof narrative templates as models. Do not invent metrics. If there is no measured outcome, describe the observed or defensible change accurately.

## Maintenance

After several scored roles, update:

- `evidence-map.md` when repeated job-description concepts do not route cleanly.
- Variant resumes when a recurring role lane needs stronger packaging.
- Proof narratives when a strong proof point is missing.
- `project-config.md` when preferences or constraints change.
- `renderer-config.json` only when document formatting behavior changes.

Keep the system evidence-bound. The goal is not to look like a fit for everything. The goal is to identify where the user is a credible fit and produce materials that make that fit clear.

## GitHub beta hygiene

Before publishing or sharing a beta copy, keep only source templates, builder files, renderer files, instructions, and fictional samples.

Do not commit:

- `__pycache__/`
- `*.pyc`
- `outputs/`
- generated DOCX files, QA reports, ZIP files, or validation caches, except the intentional fictional sample DOCX in `docs/sample-builder-outputs/`
- real user resumes, cover letters, credentials, references, proof narratives, or job-search preferences

`reference.docx` should stay in the project. It is a source template for the renderer, not a generated output.
